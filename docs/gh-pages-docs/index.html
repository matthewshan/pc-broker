Introduction

PC Broker is an internal service that sits between your homelab Kubernetes cluster and a gaming PC running Ollama. It handles power management (Wake-on-LAN wake, agent-based graceful shutdown), continuous reachability polling, a streaming chat proxy to Ollama, and an optional fail-safe idle auto-shutdown — so the PC can stay off most of the time and be woken from a phone when needed.

A lightweight built-in UI at / provides a status dashboard, power controls, and a streaming chat panel that lists whatever models Ollama has pulled.

💡 This service is internal-only. It should never be exposed to the public internet. Access is via an internal gateway or Twingate.

  └─ Ollama :11434          ← headless SYSTEM task (install-ollama.ps1)

Wake Flow

The broker never blocks a chat request on a wake. Chat requires the ready state; anything else returns 503, which the UI turns into a wake prompt: 

POST /api/llm/chat
  └─ state == ready? ──yes──► stream NDJSON from Ollama
       └─ no ──► 503 {"detail": "PC/Ollama not ready", "state": "..."}
                   └─ UI shows "Wake PC" → POST /api/power/on
                        └─ poll loop watches the PC come up
                             offline → waking → ollama_starting → ready

Operational States

The broker uses a unified state machine. The same state value appears in API responses, the UI dashboard, and logs.

offline → waking → ollama_starting → ready
any state → timeout / error
StateMeaning
offlinePC is not reachable (TCP probe on PC_REACHABILITY_PORT fails)
wakingWoL packet sent; waiting for the PC to respond
host_upLegacy — the poll loop now lands on ollama_starting/ready instead
ollama_startingPC is reachable but Ollama is not answering yet
readyPC is reachable and Ollama is healthy
timeoutWake attempt exceeded HOST_REACHABILITY_TIMEOUT
errorUnexpected error during polling

Quick Start

Run locally

The Ollama URL is derived from PC_HOST and OLLAMA_PORT unless OLLAMA_URL is set explicitly. Settings can also live in a .env file.

pip install -r requirements.txt

PC_MAC=aa:bb:cc:dd:ee:ff \
PC_HOST=192.168.1.100 \
uvicorn app.main:app --reload

Run with Docker

docker run -p 8000:8000 \
  -e PC_MAC=aa:bb:cc:dd:ee:ff \
  -e PC_HOST=192.168.1.100 \
  ghcr.io/matthewshan/pc-broker:latest

Run tests

pip install -r requirements-dev.txt
pytest tests/ -v

Power API

POST /api/power/on Send Wake-on-LAN packet ▶

Sends a UDP broadcast WoL magic packet to the gaming PC. Idempotent — safe to call even if the PC is already on.

Returns 202 Accepted on success, 503 if PC_MAC is not configured.

Response (202)

{ "accepted": true, "message": "Wake-on-LAN packet sent" }

Try it

POST /api/power/off Request graceful shutdown ▶

Calls the shutdown agent running on the gaming PC (agent/ in this repo). Returns 202 on success, 502 if the agent errors, 503 if it is not configured or unreachable.

⚠️ Requires SHUTDOWN_AGENT_URL to be configured. When API_TOKEN is set, this endpoint additionally requires an Authorization: Bearer <token> header.

Try it

LLM API

GET /api/llm/health Ollama readiness check ▶

Always returns 200 with the live Ollama reachability and the broker state. The UI polls this to decide between the chat panel and the wake prompt.

Response

{ "reachable": true, "state": "ready" }

Try it

POST /api/llm/chat Streaming chat passthrough ▶

Accepts an Ollama-format chat request and streams the response back as NDJSON (application/x-ndjson), exactly as Ollama produces it. Streaming is always on — a stream field in the request is ignored.

Requires the ready state; otherwise returns 503 {"detail": "PC/Ollama not ready", "state": "..."}, which the built-in UI turns into a wake prompt. Connection or model errors surface as 502/503 before the first streamed byte.

Request body

{
  "model": "qwen3.5:9b",
  "messages": [
    { "role": "user", "content": "Hello!" }
  ],
  "options": { "temperature": 0.7, "num_ctx": 8192 }
}

Options sanitization

Ollama honors resource-shaping options per request, and an oversized num_ctx can exhaust the PC's VRAM/RAM. Only known sampling options are passed through (temperature, top_p, top_k, min_p, seed, stop, repeat_penalty, presence_penalty, frequency_penalty, num_ctx, num_predict), with hard caps:

OptionCap
num_ctx16384
num_predict8192

Try it

Idle Auto-Shutdown

When armed with IDLE_SHUTDOWN_ENABLED=true, the broker shuts the PC down after it has been idle for IDLE_SHUTDOWN_MINUTES. It is disabled by default and designed fail-safe: every ambiguous signal is treated as "user is active" — the broker never shuts down when unsure.

A shutdown is only triggered when all of the following hold, for IDLE_CONSECUTIVE_CHECKS consecutive checks (one per IDLE_ACTIVITY_POLL_INTERVAL):

SignalCondition
Broker stateready, and past the post-wake grace period (IDLE_POST_WAKE_GRACE_MINUTES)
LLM activityNo active chat streams; last LLM activity older than IDLE_SHUTDOWN_MINUTES
User inputAgent-reported console idle time above IDLE_USER_THRESHOLD_MINUTES
GPUUtilization below IDLE_GPU_UTIL_THRESHOLD percent
Keep-awakeNot enabled via /api/idle/keep_awake

User-activity and GPU data come from the agent's /activity endpoint (agent v0.2+ with the pc-broker-idle-reporter logon task). A missing or stale report counts as active.

POST /api/idle/keep_awake Pause/resume idle auto-shutdown ▶

Toggles the keep-awake flag (also exposed in the UI). While enabled, idle auto-shutdown never triggers. Resets the current strike count.

Request body

{ "enabled": true }

Response

{ "keep_awake": true }

Gaming PC Setup

Two PowerShell installers in agent/ prepare the Windows gaming PC. Both must run from an elevated (Administrator) prompt.

Shutdown agent — install.ps1

Registers the pc-broker-agent scheduled task (runs as SYSTEM at startup) listening on port 8001. It provides the shutdown endpoint, doubles as the broker's reachability probe target, and — from agent v0.2 — exposes /activity (per-session input idle + GPU utilization) via the pc-broker-idle-reporter logon task for idle auto-shutdown. Re-running the installer stops the task and kills any lingering agent process before re-registering, so upgrades never leave a stale listener bound to the port. 

./install.ps1 -Token <SHUTDOWN_AGENT_TOKEN>

Headless Ollama — install-ollama.ps1

The standard Ollama installer registers a per-user login app, which is useless on a machine that gets woken remotely and never logged into. This script runs ollama serve as a SYSTEM scheduled task at startup, moves the model store to a shared directory, and opens the firewall port only to the hosts that need it — Ollama's API has no auth and includes destructive endpoints, so it must not be reachable from the whole LAN. 

./install-ollama.ps1 -AllowFrom 192.168.1.163 `
  -Models qwen3.5:9b,qwen3.5:4b,gemma4:e4b-it-qat
💡 Also disable Windows Fast Startup (HiberbootEnabled=0) and give the PC a static DHCP reservation — Wake-on-LAN is unreliable without both.

Kubernetes Deployment

Manifests live in the k3s-homelab repo under applications/pc-broker/ and are deployed by ArgoCD. The pod runs with hostNetwork: true — the WoL UDP broadcast never leaves the CNI overlay otherwise, so the broker must broadcast from the node's LAN interface (node and PC on the same L2 segment).

Secrets

Secrets (PC_MAC, API_TOKEN, SHUTDOWN_AGENT_TOKEN) are synced by External Secrets Operator from Infisical into the pc-broker-secrets Secret — nothing secret is stored in Git.

Container Images

Multi-arch (amd64/arm64) images are published by CI on every v*.*.* tag:
